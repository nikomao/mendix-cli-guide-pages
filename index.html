§ 01

What is mxcli? What problem does it solve?

If you've worked with Mendix Studio Pro, you already know this reality: a Mendix project is essentially a binary .mpr file. That file is human-friendly — Studio Pro visualizes everything for you — but for AI assistants and automation tools, it's an opaque black box.

mxcli is here to tear down that wall. It's an open-source CLI tool built by Mendix Labs, with one core goal: enabling AI coding assistants like Claude Code, Cursor, and Copilot to actually read and operate on Mendix projects.

🔍
MDL Query Language
SQL-like syntax for querying and describing Mendix model elements as plain text
🤖
AI Tool Integration
Auto-generates CLAUDE.md and AGENTS.md so AI understands project context
🛡️
Dev Container
Sandboxed Docker environment that limits AI agent file access, preventing accidents
✅
Starlark Linter
Custom quality rules, HTML report generation, CI/CD-ready
🧩
VSCode Extension
Project structure visualization — browse documents without opening Studio Pro
🚀
PAD Local Runtime
Spin up Mendix Runtime + PostgreSQL inside Docker containers for local testing
⚠️ Alpha Warning: mxcli is early-stage software, largely built with AI coding assistants at high velocity. Bugs exist, and it can corrupt project files. Currently validated only against Mendix 11.6. Always work on a copy or a Git branch — never directly on a production project.
§ 02

Core Architecture: MDL + Agent Integration

To understand mxcli, start with its architectural thinking.

MDL (Mendix Definition Language) is the technical heart of the tool. It provides a SQL-like syntax that lets you query the internals of a Mendix project the same way you'd query a database — inspect entities, microflows, and pages, and even modify them.

MDL — Example Statements
-- List all modules SHOW MODULES -- Inspect a specific entity DESCRIBE MyModule.Customer -- Get the MDL source of a page SOURCE MyModule/CustomerOverview -- List all microflows in a module LIST microflows FROM MyModule

Dependency Version Purpose Status

Docker Desktop Latest Dev Container and PAD runtime Required for advanced features
VS Code Latest Dev Container + Mendix extension Recommended
WSL 2 (Windows only) Windows 10/11 Required to run Docker Linux containers Required on Windows

💡 Note for Windows users: Docker on Windows requires WSL 2. If you haven't set it up yet, open PowerShell as Administrator, run wsl --install, and restart your PC. It's a one-time setup.
§ 04

Installing mxcli

mxcli is distributed as a pre-compiled binary — no Go runtime required. Just download the executable for your platform and you're done.

PowerShell
# 1. Download the latest Windows binary from GitHub Releases Invoke-WebRequest -Uri "https://github.com/mendixlabs/mxcli/releases/latest/download/mxcli-windows-amd64.exe" -OutFile "mxcli.exe" # 2. Move to a directory in your system PATH (recommended) Move-Item mxcli.exe C:\Windows\System32\mxcli.exe # Or place in a custom directory and add it to PATH $env:PATH += ";<path-to-your-tools-dir>" # 3. Verify installation mxcli --version # Expected output: mxcli version 0.x.x
zsh / bash
# 1. Download the macOS binary (Apple Silicon or Intel) # Apple Silicon (M1/M2/M3): curl -L https://github.com/mendixlabs/mxcli/releases/latest/download/mxcli-darwin-arm64 -o mxcli # Intel Mac: curl -L https://github.com/mendixlabs/mxcli/releases/latest/download/mxcli-darwin-amd64 -o mxcli # 2. Make it executable chmod +x mxcli # 3. Move to your PATH for global access sudo mv mxcli /usr/local/bin/mxcli # 4. macOS may warn "unidentified developer" — remove the quarantine flag xattr -d com.apple.quarantine /usr/local/bin/mxcli # 5. Verify installation mxcli --version
bash
# 1. Download the Linux amd64 binary curl -L https://github.com/mendixlabs/mxcli/releases/latest/download/mxcli-linux-amd64 -o mxcli # 2. Make it executable chmod +x mxcli # 3. Move to system PATH sudo mv mxcli /usr/local/bin/mxcli # 4. Verify installation mxcli --version # mxcli version 0.x.x
📦 Releases page: The latest binaries for all platforms are available at github.com/mendixlabs/mxcli/releases. If the URLs above don't match, head there to manually download the file for your platform.
§ 05

Initializing Your First Mendix Project

Once installed, the first thing to do is run mxcli init against an existing Mendix project. This command auto-generates all configuration files needed by AI tools.

Prep: Copy your project first

bash
# Always work on a copy or a Git branch! cp -r ~/Mendix/MyApp ~/Mendix/MyApp-mxcli-test cd ~/Mendix/MyApp-mxcli-test # Or use Git (recommended) cd ~/Mendix/MyApp git checkout -b feature/mxcli-experiment

Run the init command

bash
# Initialize for Claude Code (also supports --tool cursor, --tool copilot, etc.) mxcli init --tool claude "C:\Users\YourName\Mendix\MyApp" # Linux / macOS path format mxcli init --tool claude "/home/user/Mendix/MyApp" # On success you'll see output like: ✓ Created 41 skill files in .ai-context/skills/ ✓ Created .claude/settings.json ✓ Created CLAUDE.md ✓ Created 10 command files in .claude/commands/ ✓ Created 29 lint rule files in .claude/lint-rules/ ✓ Created AGENTS.md ✓ Created .devcontainer/ configuration ✓ Initialization complete!

Here's what each generated file does:

Project Directory Structure
MyApp/ ├── CLAUDE.md # Tells Claude Code how to interact with this project ├── AGENTS.md # General AI agent context document ├── .ai-context/ │ └── skills/ # 41 skill files: teaches AI how to build Mendix components ├── .claude/ │ ├── settings.json # Claude Code settings │ ├── commands/ # 10 custom slash commands │ └── lint-rules/ # 29 Mendix quality rules ├── .devcontainer/ │ └── devcontainer.json # Dev Container sandbox configuration └── MyApp.mpr # Original Mendix project file
§ 06

Querying Project Structure with MDL

mxcli gives you two ways to use MDL: single-command mode and interactive REPL mode (similar to psql or sqlplus).

📌 Single-command mode

bash
# Basic syntax: mxcli -p <project-path> -c "<MDL statement>" # List all modules mxcli -p MyApp.mpr -c "SHOW MODULES" Administration MyFirstModule System # View the document tree of a module mxcli -p MyApp.mpr -c "STRUCTURE MyFirstModule" # Inspect a specific entity's full definition mxcli -p MyApp.mpr -c "DESCRIBE MyFirstModule.Customer" # Get the MDL source of a page mxcli -p MyApp.mpr -c "SOURCE MyFirstModule/CustomerOverview_NewEdit" # List all microflows in a module mxcli -p MyApp.mpr -c "LIST microflows FROM MyFirstModule"

💬 Interactive REPL mode

REPL mode is great for continuous exploration — no need to re-specify -p on every command.

bash
# Enter REPL mode mxcli -p MyApp.mpr mxcli> SHOW MODULES Administration MyFirstModule System mxcli> LIST pages FROM MyFirstModule CustomerOverview CustomerDetail Home_Web mxcli> DESCRIBE MyFirstModule.Order Entity: Order Attributes: - OrderId : AutoNumber - Status : Enum (New, Processing, Shipped) - TotalAmount : Decimal - CreatedDate : DateTime Associations: - Order_Customer → Customer (ManyToOne) mxcli> .exit
§ 07

Running Lint Checks and Quality Reports

mxcli ships with a built-in Starlark-based linter that checks for common Mendix model issues and generates HTML quality reports.

bash
# Run lint check (output to terminal) mxcli lint -p MyApp.mpr Checking module: MyFirstModule ⚠ MF001: Microflow 'ACT_Customer_Create' has no error handling ⚠ PG002: Page 'CustomerOverview' has more than 20 data grid columns ✓ ENT003: All entities have documentation Lint complete: 2 warnings, 0 errors # Generate an HTML report (easy to share with your team) mxcli report -p MyApp.mpr --format html --output report.html ✓ Report generated: report.html # Lint only a specific module mxcli lint -p MyApp.mpr --module MyFirstModule
✅ CI/CD tip: Add mxcli lint -p yourapp.mpr to your CI pipeline to automatically check Mendix model quality before every merge. A non-zero exit code signals errors.
§ 08

Collaborating with Claude Code

This is the most exciting part of mxcli. The CLAUDE.md file generated by mxcli init lets Claude Code understand the full project context the moment it starts up.

bash
# 1. Launch Claude Code inside the project directory cd ~/Mendix/MyApp claude # Claude Code automatically reads CLAUDE.md and learns the available MDL commands. # You can then give it natural language instructions, for example: You: Show me all entities in MyFirstModule, then generate a list page for the Customer entity. Claude: Let me explore the project structure with mxcli first... $ mxcli -p MyApp.mpr -c "LIST entities FROM MyFirstModule" Found: Customer, Order, Product, Category Now I'll generate a list page for the Customer entity...
🧠 How it works: After reading CLAUDE.md, Claude Code knows it can call mxcli commands to search project structure and fetch MDL source. Combined with the skill files in .ai-context/skills/, it generates changes that follow Mendix best practices. The AI isn't guessing your project structure — it's actually reading it.

A real-world example from the community: in a few hours, Claude Code used mxcli to build a complete business system with 12 entities, 25 microflows, and 17 admin pages — entirely without opening Studio Pro.

§ 09

Advanced: Dev Container Sandbox

If you plan to regularly let an AI agent autonomously modify your project, the Dev Container mode is strongly recommended. It uses Docker to isolate the agent's file system access, preventing accidental changes outside your project directory.

bash
# Prerequisite: make sure Docker Desktop is running # 1. Open the project in VS Code (the .devcontainer/ config is already there) code ~/Mendix/MyApp # 2. VS Code detects .devcontainer/ and shows a prompt. # Click "Reopen in Container" to enter the sandbox. # 3. Once the container starts, mxcli is pre-installed. # Use it directly from the container's terminal. root@container:~# mxcli --version mxcli version 0.x.x # 4. (Optional) Start the Mendix Runtime via PAD for local testing mxcli start -p MyApp.mpr ✓ Starting Mendix Runtime on port 8080... ✓ PostgreSQL ready on port 5432 ✓ Application available at http://localhost:8080
🔒 Why the sandbox matters: The Dev Container restricts the AI agent's filesystem access to the mounted project directory. Even if the AI generates unexpected file operations, your host system stays untouched — only files inside /workspaces/MyApp can be modified.
§ 10

Current Limitations & Pitfall Guide

As an alpha-stage tool, mxcli has some clear boundaries you should understand:

Feature Status Recommendation
Reading / exploring project structure ✅ Stable Use freely — this is the most mature capability
Lint quality reports ✅ Stable Ready to integrate into CI/CD
MDL as a query language ✅ Core innovation Syntax still evolving — watch the Changelog
AI agent integration concept ⚠️ Experimental Great for exploration, not production development
Writing / modifying project files ⚠️ Use with caution Always work on a Git branch or copy
Mendix version compatibility ⚠️ Only 11.6 validated Other versions may work, but expect rough edges
Dev Container + PAD ⚠️ Requires Docker Windows users must configure WSL 2 first
🛑 The Golden Rule: Before any write operation with mxcli, commit your Git changes first. If your project file gets corrupted, git checkout -- . is your last line of defense. And never — under any circumstances — run mxcli directly against a production project.
