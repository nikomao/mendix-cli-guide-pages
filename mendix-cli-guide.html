01 / WHY

为什么 Mendix 需要 mxcli

`.mpr` 对 Studio Pro 友好，对 AI 和自动化流程不友好。mxcli 的价值，是把模型结构转换成可查询、可解释、可接入 Agent 的文本接口。

02 / WHAT

它核心做了什么

提供 MDL 查询语言、AI 工具初始化、Lint 审计、HTML 报告，以及 Dev Container + PAD 这种更安全的实验环境。

03 / HOW

正确的落地顺序

先准备副本和 Git 分支，再安装 mxcli，接着跑 `mxcli init`，最后才把它接入 Claude Code、Copilot 或 Cursor。

04 / RISK

当前最值得警惕的点

不要一上来让 AI 写入生产项目。先用查询和报告建立信心，再逐步尝试页面、实体、微流级别的实验性改动。

§ 01

mxcli 是什么，它到底解决了什么问题

一句话概括：它让 AI 编程助手不再“猜” Mendix 项目，而是真正“读” Mendix 项目。

Mendix Studio Pro 的开发体验建立在可视化建模之上，但 `.mpr` 本质上是二进制项目文件。对开发者来说这不一定是问题；但对 AI Agent、脚本化流水线和命令行工作流来说，这是一层天然黑盒。

mxcli 的意义，在于把这层黑盒撬开。它通过 MDL 让项目结构变成可查询的文本世界，再通过 `CLAUDE.md`、`AGENTS.md` 和技能文件，让 Agent 明白项目上下文、约束和操作路径。

🔍 MDL 查询语言

类 SQL 语法，支持模块、实体、页面、微流等结构查询，是 AI 理解 Mendix 模型的入口。

🤖 Agent 初始化

运行 `mxcli init` 后，会生成适配 Claude Code 等工具的上下文文件和技能目录。

🧪 Lint 与报告

内置 Starlark 规则，可在终端输出问题，也能生成 HTML 报告供团队共享。

🛡 沙箱式开发

结合 Dev Container 和 PAD，本地可以在更隔离的容器环境里试验 Agent 改动。

§ 02

核心架构：MDL + Agent 集成

理解 mxcli，最有效的方式不是记命令，而是先看它如何把 `.mpr` 接成 AI 可用接口。

Mendix Project → mxcli

`.mpr` 文件进入 mxcli 后，被 MDL 引擎和目录表解析成可查询的结构信息。

mxcli → Agent Context

初始化后，CLAUDE.md、AGENTS.md 和技能目录共同组成 Agent 的执行上下文。

mxcli → REPL / CLI

你既可以脚本化跑单条查询，也可以进入 REPL 连续探索结构。

mxcli → Lint / HTML Report

质量规则和报告输出让它不止是“查询工具”，也是治理工具。

MDL 示例语句 
SHOW MODULES
DESCRIBE MyModule.Customer
SOURCE MyModule/CustomerOverview
LIST microflows FROM MyModule
STRUCTURE MyModule
§ 03

前置要求：安装前先把环境边界想清楚

这一步决定后续体验是否顺畅，尤其是 Windows 用户和准备尝试 Dev Container 的团队。

依赖项 推荐版本 用途 建议级别
Mendix Studio Pro 11.6 `.mpr` 文件来源，也是当前公开验证最明确的版本线 必须
Git 任意新版本 保护项目文件，回滚实验性改动 强烈推荐
VS Code 最新版 承载 Dev Container、扩展和 AI Agent 工作流 推荐
Docker Desktop 最新版 支持 Dev Container 和 PAD 本地运行时 进阶功能需要
WSL 2 Windows 10/11 Windows 上运行 Linux 容器的基础 Windows 必须
Claude Code / Copilot 最新版 承接 Agent 协作 按需安装
💡 Windows 额外注意

如果你计划使用 Docker 或 Dev Container，先执行 `wsl --install` 并重启。很多“容器起不来”的问题，不是 mxcli 本身的问题，而是底层 WSL 2 环境没就绪。

§ 04

安装 mxcli

mxcli 以预编译二进制形式分发，不需要本地准备 Go 运行时，直接下载即可。

🪟 Windows
Step 1 · Download 
Invoke-WebRequest -Uri "https://github.com/mendixlabs/mxcli/releases/latest/download/mxcli-windows-amd64.exe" -OutFile "mxcli.exe"
Step 2 · Move to PATH 
Move-Item mxcli.exe C:\Windows\System32\mxcli.exe
Step 3 · Verify 
mxcli --version
🍎 macOS / 🐧 Linux
macOS
Step 1 · Download 
curl -L https://github.com/mendixlabs/mxcli/releases/latest/download/mxcli-darwin-arm64 -o mxcli
Step 2 · Make executable 
chmod +x mxcli
Step 3 · Install and trust 
sudo mv mxcli /usr/local/bin/mxcli
xattr -d com.apple.quarantine /usr/local/bin/mxcli
Linux
Step 1 · Download 
curl -L https://github.com/mendixlabs/mxcli/releases/latest/download/mxcli-linux-amd64 -o mxcli
Step 2 · Make executable 
chmod +x mxcli
Step 3 · Move to PATH 
sudo mv mxcli /usr/local/bin/mxcli
安装提示

如果直链未来变化，统一回到 GitHub Releases 页面获取最新二进制即可。macOS 上首次运行未签名二进制时，记得处理 quarantine 标记。

§ 05

初始化第一个 Mendix 项目

真正关键的第一条命令不是查询，而是先运行 `mxcli init`。

先隔离风险

先在副本或 Git 分支中操作。这个习惯比任何命令都重要。

创建实验副本 
cp -r ~/Mendix/MyApp ~/Mendix/MyApp-mxcli-test
cd ~/Mendix/MyApp-mxcli-test
或切换到 Git 分支 
git checkout -b feature/mxcli-experiment
生成 Agent 上下文

初始化时指定 AI 工具，生成上下文文件和技能目录。

Windows 示例 
mxcli init --tool claude "C:\Users\YourName\Mendix\MyApp"
macOS / Linux 示例 
mxcli init --tool claude "/Users/you/Mendix/MyApp"
§ 06

用 MDL 查询项目结构

初始化完成后，mxcli 就成为你探索 Mendix 项目的命令行入口。

单行命令模式

适合脚本化、适合复制进 Agent 执行链，也适合 CI 或文档演示。

列出模块 
mxcli -p MyApp.mpr -c "SHOW MODULES"
查看实体 
mxcli -p MyApp.mpr -c "DESCRIBE MyFirstModule.Customer"
查看页面源码 
mxcli -p MyApp.mpr -c "SOURCE MyFirstModule/CustomerOverview"
REPL 交互模式

适合连续探索，在你还不知道下一条要查什么时更顺手。

REPL 
mxcli -p MyApp.mpr

mxcli> SHOW MODULES
mxcli> LIST pages FROM MyFirstModule
mxcli> DESCRIBE MyFirstModule.Order
mxcli> .exit
推荐实践

不要一开始就让 Agent 生成页面。先跑 `SHOW`、`DESCRIBE`、`SOURCE` 等查询命令，确认它拿到的是正确结构，再放开下一步。

§ 07

运行 Lint 和质量报告

如果只想先低风险试用 mxcli，Lint 是比“直接写项目”更稳的切入口。

终端检查 
mxcli lint -p MyApp.mpr
生成 HTML 报告 
mxcli report -p MyApp.mpr --format html --output report.html
只检查单个模块 
mxcli lint -p MyApp.mpr --module MyFirstModule
团队引入建议

先把 `mxcli lint` 接进 CI，可以先消费“读取”和“审计”能力，而不是一下子把写入改动交给 Agent。

§ 08

与 Claude Code、Copilot 等 Agent 协同开发

这是 mxcli 与普通 CLI 最大的差异化，也是最值得关注的部分。

初始化后，Agent 不再需要纯靠目录结构和文件名猜测你的项目。它可以读取上下文文件，再调用 mxcli 查询实体、页面、微流和源码片段，从而在执行任务前形成更可靠的理解。

进入项目并启动 Agent 
cd ~/Mendix/MyApp
claude
给出自然语言任务 
请先列出 MyFirstModule 下的 Entity，
再根据 Customer 实体生成一个列表页面，
过程中先展示你查询到的结构。
风险边界

即便 Agent 已经能查结构，也不要默认它的写入一定安全。让它先展示查询结果和行动计划，再决定是否放行修改，会更稳。

§ 09

进阶：Dev Container 沙箱环境

如果你真的打算让 Agent 持续改项目，容器隔离不是可选项，而是风险控制手段。

Dev Container 的意义，是把 Agent 的文件访问范围限制在项目目录，避免误操作污染主机环境。对团队来说，这比单纯依赖“大家小心点”更可靠。

打开项目 
code ~/Mendix/MyApp
进入容器后验证并启动 
mxcli --version
mxcli start -p MyApp.mpr
PAD 本地运行

PAD 可以在 Docker 容器内启动 Mendix Runtime + PostgreSQL，用于本地测试。这意味着你可以在更可控的环境里验证 Agent 生成或修改后的效果。

§ 10

当前限制与避坑指南

这部分不该放在最后被忽略，反而应该在真正动手前先看一遍。

能力 当前判断 建议使用方式
读取 / 探索结构 稳定 优先使用，这是理解项目和驱动 Agent 的基础。
Lint / 报告 稳定 非常适合先接入团队流程。
MDL 查询 核心能力 建议先用它建立结构认知，再做其它动作。
AI 自动写入 实验阶段 必须在副本或 Git 分支上操作，并保留人工复核。
版本兼容性 11.6 最明确 其它版本先小范围验证，不要直接全量推广。
生产项目直改 不建议 即使是只读查询，也建议先在可回滚环境里建立流程。
黄金规则

在任何写入操作前，先提交 Git 变更。项目损坏时，版本控制是最后一道防线。不要把“我只是试一下”当作放弃隔离和备份的理由。

如果你是第一次把 Mendix 接入 AI 工作流，这页已经够你起步了

建议不要一次把所有能力都开起来。先完成安装、初始化和结构查询，确认团队对 mxcli 的边界理解一致，再决定是否推进到 Dev Container、PAD 和写入型 Agent 协作。
